第 3 课

一次发送任务的完整调用链

这是最重要的一条链。理解它，后面改功能就不会乱。

1. 前端用户点击发送，进入 runTask()frontend/src/main.tsx
2. HTTP请求 POST /api/chat/streambackend/main.py
3. 记忆先追加用户输入 user/input/messageMemoryStore.append()
4. 编排创建并执行 AgentGraph.run()backend/agents/graph.py
5. 路由管理者分类后决定调用哪些智能体_classify() / _flow_for()
6. 执行Coding 通过工具写文件，Verifier 跑测试_do_action()
7. 返回后端持续返回 NDJSON 事件和最终结果StreamingResponse
8. 展示前端把 result 格式化为 Agent 消息formatResultMessage()
前端发送任务的核心代码

第 4 课

后端入口：FastAPI 只负责进出

backend/main.py 不是业务核心，它负责接口、会话、目录选择和流式响应。

模型接口

/api/models 读取和保存模型配置，/api/models/{model_id}/test 测试连通性。

会话接口

/api/sessions 创建会话，/api/history 恢复历史，删除接口只删除 memory。

项目选择

/api/pick-dir 在 macOS 用 osascript，Windows 用 PowerShell，失败时尝试 tkinter。

流式任务

/api/chat/stream 用后台线程跑 AgentGraph，用队列把事件交给 StreamingResponse。

chat_stream 的职责
def chat_stream(req: ChatRequest) -> StreamingResponse:
    event_queue = queue.Queue()
    workdir = _session_workdir(req.session_id)
    memory_store.append(workdir, req.session_id, "user", "input", "message", req.text)

    def worker() -> None:
        graph = AgentGraph(model_store, memory_store, emit=emit)
        result = graph.run(...)
        event_queue.put({"type": "result", "data": result.model_dump()})

    threading.Thread(target=worker, daemon=True).start()
    return StreamingResponse(stream(), media_type="application/x-ndjson")
第 5 课

AgentGraph：整个项目的编排核心

这里决定了普通问候不会读仓库，代码任务才会进入 Coding，文档任务才会进入 Doc。

manager()

检查 pending Plan、saved Plan、中断状态，构造 Context Package，并决定第一跳。

_classify()

先用 Python 规则识别问候、代码、文档、解释、修改任务，兜底才调用模型。

_flow_for()

把任务类型变成 LangGraph 路由，例如 repo -> coder -> verifier -> doc。

planner()

负责 Plan 选择题、保存 pending Plan、生成计划文件并等待用户确认。

coder()

ReAct 循环调用模型，读取 actions，再通过工具真实写文件或跑命令。

verifier()

按项目结构选择测试命令，失败时最多回到 Coding 修复一次。

管理者的核心分支
if pending_plan and self._is_plan_cancel(state["text"]):
    route = "final"
elif pending_plan and self._should_treat_as_plan_reply(state["text"], pending_plan, state):
    state["text"] = self._compose_pending_plan_text(pending_plan, reply)
    route = "planner"
elif saved_plan and self._is_execute_plan_text(state["text"]):
    state["text"] = self._compose_saved_plan_text(saved_plan, state["text"])
    route = "repo"
else:
    classification = self._classify(state)
用户输入推荐路由说明
你好manager -> final普通对话，不读仓库，不写文件。
解释这个报错manager -> repo -> answer -> final需要仓库上下文，但不修改代码。
生成 READMEmanager -> repo -> doc -> final文档任务不走 Coding。
创建一个系统manager -> repo -> coder -> verifier -> final代码任务必须真实写磁盘并验证。
创建系统并写文档manager -> repo -> coder -> verifier -> doc -> final验证通过后再生成文档。
第 6 课

Memory：会话能继续的原因

历史会话、Plan 上下文、中断恢复都不是前端状态，而是 memory 目录里的文件。

memory/ config/ models.json data/ global.md projects/ <项目路径哈希>/ project.md sessions/ <会话id>.jsonl

会话记录

每条 jsonl 记录包含 id、ts、ag、tl、k、out、m。

历史恢复

list_history() 扫描项目 memory，只提取用户输入和管理者最终结果展示到对话窗口。

Plan 上下文

Plan 提问后写入 pending_plan，用户回复 1A, 2A 时管理者能找回原始需求。

中断判断

如果最后一条不是 manager/final/result，系统认为上次可能异常中断。

会话记忆的一条记录
{
  "id": 1,
  "ts": "2026-07-03T00:00:00+00:00",
  "ag": "manager",
  "tl": "classify",
  "k": "context",
  "out": "任务分类：code_gen",
  "m": {}
}
第 7 课

前端工作台：状态和展示分清楚

前端负责交互和展示，不负责复杂编排。真正可恢复的历史来自 memory。

状态用途常见问题
models模型列表新增模型不显示时看 loadModels()。
workdir当前项目目录未选择项目时不能新建会话。
sessionId当前会话 id历史恢复时必须同时恢复 workdir。
events右侧事件流看 NDJSON 的 event 是否解析成功。
messages中间对话窗口最终回复由 TaskResult 格式化。
planModePlan 模式开关只影响请求参数，不直接生成计划。

消息气泡

用户和 Agent 使用同一种结构：.message-row 包住 .bubble，复制按钮放在气泡外部右下角。

历史弹窗

历史项目和会话来自 /api/history。删除按钮只删除 memory，不删除真实项目目录。

消息结构
<div className={`message-row ${msg.role}`}>
  <article className={`bubble ${msg.role}`}>
    <span>{msg.role === "user" ? "用户" : "Agent"}</span>

第 8 课

上手修改功能：改哪里、怎么验

这一节是以后改功能时的速查表。

改任务分类

优先改 AgentGraph._classify() 和 _flow_for()，必要时同步改 MANAGER_PROMPT。

改 Plan 模式

看 planner()、_latest_pending_plan()、_build_plan_reply()、_compose_pending_plan_text()。

新增智能体

改 api/schema.py、前端 agents、prompts.py、graph.py 和测试。

新增 Coding 工具

改 backend/tools/、_do_action()、CODER_PROMPT，并测试路径安全。

改验证命令

改 verifier() 和 tests/test_verifier.py，例如增加 Ruff 或前端构建。

改消息样式

改 frontend/src/main.tsx 的消息结构和 style.css 里的 .bubble、.copy-message。

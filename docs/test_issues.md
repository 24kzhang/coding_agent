# Coding Agent 实战测试问题记录

本文记录使用本项目自身的 Plan-Execute + ReAct 流程，在健身动作网站实战中暴露的问题。每项只保留现象、原因、解决方法和验证方式，便于后续回归。

## 一、Agent 编排与执行问题

### 1. 执行计划再次进入 Plan

- 现象：用户回复“执行计划”后，管理者再次调用 Plan 智能体，没有进入仓库读取和 Coding。
- 原因：管理者只判断了当前是否开启 Plan 模式，没有区分“生成计划”和“执行已保存计划”两个阶段。
- 解决：为状态增加 `executing_plan`，识别执行意图后关闭本轮 Plan 路由，直接进入执行流程。
- 验证：计划确认后事件流应依次进入管理者、仓库读取、Coding，而不是再次提问。

### 2. 计划过早标记为已执行

- 现象：Coding 或测试失败后，会话记忆仍可能显示计划已执行。
- 原因：完成状态在真正验证成功前写入。
- 解决：只在 Coding 成功且验证全部通过后写入计划完成状态；失败时保留可重试状态。
- 验证：失败会话不得出现 `plan_executed`，成功会话才允许出现。

### 3. 大型媒体目录拖垮仓库读取

- 现象：读取包含上千张图片和 GIF 的项目时，仓库索引体积过大，模型上下文被无关文件占用。
- 原因：仓库读取未区分源码、文本数据与二进制媒体文件。
- 解决：索引阶段过滤图片、GIF、字体、压缩包、数据库和系统文件，并限制文本片段预算。
- 验证：仓库摘要只列出相关代码、配置和必要数据说明，不读取媒体二进制内容。

### 4. ReAct 重复读取和重复执行

- 现象：Coding 反复调用 `list_files`、读取同一文件或运行相同命令，消耗步骤却没有推进任务。
- 原因：执行循环没有动作签名去重，也没有停滞检测。
- 解决：记录动作签名，拒绝无变化的重复读取和重复命令；连续无有效进展时结束本轮并交给验证器反馈。
- 验证：事件流应明确提示重复动作被拒绝，Coding 必须改用写入、精确修改或测试等推进动作。

### 5. 重试时使用旧文件内容覆盖新代码

- 现象：验证失败后的下一轮 Coding 可能基于旧上下文整文件重写，覆盖上一轮已经正确的修改。
- 原因：重试上下文没有强制重新读取磁盘上的最新文件。
- 解决：重试时附带当前变更文件，并要求修改前重新读取；文件写入后清除旧读取签名。
- 验证：修改保护应拒绝未读取当前版本的写操作，重新读取后才能继续。

### 6. Coding 自报完成但工具动作未成功

- 现象：模型输出完成信号，但写文件、替换或命令实际失败，流程仍可能进入成功分支。
- 原因：成功判断只依赖模型文本，没有绑定工具执行结果。
- 解决：将模型完成信号与 `coding_ok` 合并判断；存在工具失败或未完成必要写入时不得通过。
- 验证：失败工具动作必须进入验证或重试，不能直接生成成功回复。

### 7. 失败总结引用旧回复

- 现象：最终返回的失败原因与当前 Coding 或验证器的真实错误不一致。
- 原因：总结优先使用状态中残留的旧 `final` 字段。
- 解决：失败时优先采用当前错误、Coding 摘要和验证结果，旧回复只作最低优先级兜底。
- 验证：前端最终消息应直接展示本轮最近一次失败原因。

## 二、生成项目暴露的问题

### 8. SSE 事件格式错误

- 现象：服务端只使用单个换行结束事件，浏览器可能无法及时解析流式消息。
- 原因：混淆了普通文本行与 SSE 事件边界。
- 解决：每个 `data:` 事件使用两个换行结束，结束事件同样使用标准格式。
- 验证：接口响应应为 `text/event-stream`，客户端可逐事件收到内容并识别 `[DONE]`。

### 9. 首次聊天上下文没有保存

- 现象：第一次提问能返回结果，第二次提问失去动作信息和此前对话。
- 原因：首次请求只传动作对象，后端却只在单独存在 `exerciseId` 时写入缓存。
- 解决：首次请求从动作对象中取得 ID，建立服务端上下文并保存用户、助手消息；后续请求按 ID 读取。
- 验证：第二次请求只传动作 ID 时，模型消息仍包含上一轮对话上下文。

### 10. 前后端接口结构不一致

- 现象：后端返回 `{total, exercises}`，前端却把整个响应当作数组调用 `length` 和 `filter`。
- 原因：接口返回结构在前后端分别实现，没有契约测试。
- 解决：统一返回结构，并让接口测试和前端逻辑共同验证该结构。
- 验证：首页应正确显示 1324 条数据，筛选时不产生 JavaScript 类型错误。

### 11. 后端调用不存在的模型方法

- 现象：聊天接口运行时调用 `stream_chat`，但客户端只实现了 `chat_stream`。
- 原因：后端和模型封装层在不同轮次被整文件重写，方法契约发生漂移。
- 解决：统一方法名和参数签名，并增加使用模拟模型流的聊天接口测试。
- 验证：测试应实际执行聊天生成器，而不只检查响应头。

### 12. HTML 类名与 CSS 选择器不匹配

- 现象：动作卡片能生成，但图片、标题和标签样式不生效。
- 原因：JavaScript 使用 `card-*` 类名，CSS 定义的是 `exercise-card-*`。
- 解决：统一类名；静态检查同时比对 HTML 和 JavaScript 中使用的类名是否存在于 CSS。
- 验证：桌面浏览器中卡片布局、图片尺寸、标签和悬停状态均正常。

### 13. 测试代码自身无法编译

- 现象：`pytest` 收集阶段报错，`b'健身动作'` 触发非 ASCII bytes 字面量语法错误。
- 原因：测试生成后没有先做 Python 编译检查。
- 解决：使用 UTF-8 字符串解码后断言，或把期望文本编码为 bytes；验证器先运行 `compileall` 再运行测试。
- 验证：测试文件可以正常收集，全部断言实际执行。

### 14. 测试路由和真实路由不一致

- 现象：测试请求 `/api/exercises/<id>`，实现提供 `/api/exercise/<id>`，测试必然返回 404。
- 原因：测试没有依据最终代码中的真实接口生成。
- 解决：重试时把当前变更文件提供给 Coding；语义验证器比对测试、实现和前端调用三方契约。
- 验证：详情接口测试使用与前端完全一致的 URL。

### 15. 缺少失败兜底和大列表优化

- 现象：GIF 失败时出现破图；SSE 暂时失败后只能手动重发；一次渲染 1324 个卡片可能卡顿。
- 原因：实现只覆盖理想路径，没有落实计划中的异常处理和分页要求。
- 解决：增加图片加载失败占位、有限次数 SSE 重试，以及分页或分批渲染。
- 验证：无效图片路径有明确占位；模拟首次 SSE 失败后可重试；首屏只渲染当前页数据。

### 16. LongCat 在 JSON 模式下连续返回空动作

- 现象：Coding 多轮收到 `{}`，没有工具动作，也没有完成信号；换其他模型后仍可能表现为结构化输出不稳定。
- 原因：编排层把所有供应商都强制成同一种 `response_format=json_object` 协议，没有使用 LongCat 稳定支持的原生工具标签。
- 解决：模型名包含 LongCat 时，Coder 从第一轮就使用普通文本请求，并追加 `LONGCAT_CODER_SUFFIX`；`LlmClient` 把 `<longcat_tool_call>` 标签转换为统一 actions。
- 验证：`test_coder_uses_plain_text_protocol_for_longcat_immediately` 和原生标签解析测试通过；真实任务第一轮即可得到工具动作。

### 17. 长文件读取被截断，模型无法看到待修改位置

- 现象：HTML、CSS 或测试文件超过 12,000 字符时，模型反复读取或基于半截内容重写。
- 原因：默认读取上限和当前文件快照预算过小，且工具观察与文件内容混在一起反复滚动。
- 解决：初始源码读取规范化为至少 24,000 字符；单文件快照上限 24,000、全部快照预算 48,000，工具结果另行压缩。
- 验证：`test_normalize_actions_expands_initial_source_read` 和 Coder 上下文测试通过，真实 CSS 修复能读取完整目标文件。

### 18. LongCat 大工具响应在非流式请求中超时

- 现象：模型已经生成较长文件内容，但客户端等待完整响应时超时，重试会重复消耗额度。
- 原因：模型封装对 LongCat 仍使用一次性 JSON 响应，没有按供应商 SSE 能力读取增量内容。
- 解决：LongCat 自动发送 `stream=true`，逐行拼接 `delta.content`；忽略 usage、心跳和空 `choices`，同时设置总时长和 100,000 字符正文上限。
- 验证：`test_longcat_chat_streams_and_joins_sse_content` 通过；真实长 CSS 工具调用完成而未被误判超时。

### 19. 工具动作缺少必填参数时错误不明确

- 现象：模型返回 `read_file` 却缺少 `path`，上层只看到通用异常，下一轮不知道该修正哪个字段。
- 原因：工具分发前没有统一参数校验。
- 解决：`_do_action()` 对不同工具逐项检查必填字段，返回可观察的中文错误；Coder 将其作为权威失败 observation 反馈给模型。
- 验证：动作参数单元测试通过，真实任务能在错误动作后继续产生新动作。

### 20. Shell 中的 Git 命令读取了父仓库

- 现象：在尚未 `git init` 的测试项目中运行 `git diff --stat`，结果却来自 Coding Agent 自身仓库。
- 原因：Git 默认向父目录寻找 `.git`；此前只有 `GitTool` 做了边界保护，`run_command` 仍可绕过。
- 解决：`ShellTool` 在所选目录没有独立 `.git` 时，仅允许 `git init`；同时 `uv init` 自动加 `--no-workspace`，外部 `VIRTUAL_ENV` 不再传给用户项目。
- 验证：`test_shell_git_command_does_not_walk_into_parent_repository`、uv workspace 和虚拟环境测试通过。

### 21. 只跑后端测试没有发现页面样式接线错误

- 现象：接口和 20 个测试全部通过，但浏览器中的卡片几乎没有样式。
- 原因：HTML/JavaScript 使用的 class 与 CSS 选择器不一致，旧静态检查只核对 DOM id 和内联函数。
- 解决：验证器增加 HTML/JS/CSS class 接线检查；真实浏览器仍作为桌面项目最终验收步骤。
- 验证：修复后 1440×900 首页卡片、详情弹窗、筛选和分页正常，浏览器控制台 0 错误、0 警告。

### 22. 静态 DOM 测试误判运行时动态节点

- 现象：新增回归测试要求 `typing-indicator` 静态存在于 HTML，但该节点实际由 JavaScript 在发送消息时创建，导致 1 个错误失败。
- 原因：测试没有区分静态依赖 id 与运行时动态 id。
- 解决：拆分 `APP_JS_REQUIRED_IDS` 和 `APP_JS_DYNAMIC_IDS`，静态 HTML 只验证前者，动态节点在 JavaScript 契约中单独记录。
- 验证：Agent 自主修复后生成项目 24 个测试全部通过，语义验证器复核通过。

### 23. 生成的测试本身出现语法错误

- 现象：Agent 新增 favicon 测试时在普通函数中使用 `await`，pytest 收集阶段失败。
- 原因：一次写入同时修改实现和测试，模型没有先执行编译检查。
- 解决：Coder 根据 `compileall`/pytest 反馈精确修改测试；验证器固定先编译再运行测试。
- 验证：最终 `compileall`、`uv sync --dry-run` 和 `uv run pytest -q` 均通过。

### 24. 生成项目功能通过但不符合主仓库 Ruff 规则

- 现象：生成项目 24 个测试通过，但主仓库执行 `ruff check .` 仍报告旧式 `Optional`、多余文件模式、未使用导入和导入顺序问题。
- 原因：生成项目验证命令没有包含 Ruff，而主仓库的全量 Ruff 会递归检查 `run_test/`。
- 解决：只做等价的机械修正，使用 `str | None`、移除无用导入并整理导入顺序，不改变接口和测试语义。
- 验证：主仓库 Ruff 全部通过，生成项目再次独立运行仍为 `24 passed`。

### 25. macOS 启动脚本没有执行权限

- 现象：`start.sh` 内容正确，但直接执行可能返回 `Permission denied`。
- 原因：Agent 写入文本文件时不会自动设置 POSIX 可执行位。
- 解决：交付验收时为 `start.sh` 添加执行权限；主 Agent 的 `start.command` 同样保持可执行。
- 验证：两个脚本均通过 `bash -n`，文件权限均包含执行位。

### 26. 文件修改后被迫重复读取

- 现象：同一文件每次精确修改后都要再次调用模型读取，工具轮次和上下文持续膨胀。
- 原因：Coder 写入后删除了文件快照和可修改标记，把安全校验错误实现成了强制重读。
- 解决：写入成功后由编排器直接从磁盘刷新快照，并保持文件可修改状态。
- 验证：连续两次精确修改只需要一次初始读取，第二次修改看到第一次落盘后的真实内容。

### 27. LongCat 生成超长工具参数导致五分钟超时

- 现象：模型尝试把长函数或完整文件放进工具参数，单轮持续到 300 秒上限。
- 原因：Coding 调用没有独立输出预算，超时后还会原样携带所有已读文件快照。
- 解决：Coding 单轮限制 6,000 输出 token；超时后只保留任务明确点名的文件快照。
- 验证：请求体携带 `max_tokens=6000`，超时收缩上下文和恢复动作测试通过。

### 28. 精确替换协议要求复制整段旧函数

- 现象：`replace_file` 的 `old` 参数包含整个长函数时，工具标签在输出上限处被截断。
- 原因：工具只有全文写入和 old/new 精确替换，缺少长代码块编辑能力。
- 解决：新增 `replace_block`，用唯一开始标记、唯一结束标记和新代码块完成替换；空结束标记表示到 EOF。
- 验证：真实 Agent 成功修改 13,399 字符的 `static/app.js`，随后 `node --check` 通过。

### 29. 语义验证器把提示词截断误认为磁盘文件损坏

- 现象：13,399 字符的 JavaScript 已通过语法检查，验证模型却报告文件在 12,000 字符处被截断。
- 原因：验证器固定只读取每个变更文件前 12,000 字符，却没有标明这是上下文裁剪。
- 解决：在 60,000 字符总预算内，单个普通源码最多完整读取 48,000 字符；真正裁剪时加入不可误判的说明。
- 验证：大于 12,000 字符的合法源码尾部进入审查提示，最终语义验收通过。

### 30. 只读验收失败错误回流 Coding

- 现象：用户明确要求禁止修改，验证缺少一条命令时仍调用 Coding。
- 原因：`route_after_verifier()` 没有区分代码修复任务和 `verify` 只读任务。
- 解决：`verify` 类型无论成功或失败都直接进入 Final；静态 JavaScript 项目自动运行 `node --check`。
- 验证：只读失败路由测试通过，真实验收只调用管理者、仓库读取和验证智能体。

### 31. 管理者只识别少量固定只读短语

- 现象：“只读验证，禁止修改文件”被误分类为答疑任务。
- 原因：规则只覆盖“只验证”“不要修改文件”等精确字符串。
- 解决：把验证意图与只读限制组合判断，覆盖“只读验证”“只读验收”“禁止修改文件”等表达，同时保留“不要修改其他文件”的写入范围语义。
- 验证：三组自然表达分类回归通过，真实请求稳定进入 verifier。

### 32. 本轮无法重新拉起 GUI 浏览器

- 现象：Playwright CLI 启动 Chrome 时先遇到用户缓存权限，调整到临时缓存后又被 Codex 桌面额度审批拒绝。
- 原因：这是当前 Codex 执行环境的 GUI 权限限制，不是 Agent 或生成网站的运行错误。
- 解决：不绕过审批；改用 Node 内置 `vm` 加载真实 `static/app.js`，模拟 DOM 和 SSE 完成关键状态验收。
- 验证：首次失败回滚、首次重试成功、后续失败回滚均通过；此前 1440×900 桌面浏览器的布局、筛选、分页、详情与控制台验收结果仍保留。

## 三、最终验证结论

- 成功会话：`0483f33cb845`，所有智能体继续使用 `LongCat-2.0`，没有通过切换模型规避编排问题。
- 生成项目：`/Users/zhang/learning/coding_agent/run_test/exercises`。
- 自动验证：生成项目 `node --check`、`compileall`、`uv sync --dry-run` 通过，`uv run pytest -q` 为 `24 passed`；主项目 Ruff、90 个 Python 测试、前端类型检查和生产构建全部通过。
- 数据与接口：共 1,324 条动作、每页 24 条；“腰部 + 自重”组合筛选返回 89 条、4 页；详情图片和 GIF 可正常访问。
- 浏览器验收：1440×900 首页、组合筛选、分页、详情弹窗和聊天入口可用，控制台 0 错误、0 警告。
- 结论：本轮问题主要来自模型协议适配、ReAct 上下文、工具边界和验证覆盖，而不是 LongCat 本身；修复均落在通用编排或模型封装层，并有回归测试保护。
